# Feature Specification: 主命令框架与文本替换子命令 (trt)

**Feature Branch**: `001-text-replace-tool`

**Created**: 2026-06-01

**Status**: Draft

**Input**: User description: "主命令支持 -ls 列出子命令注释、-h 输出帮助、无子命令时等同 -ls；第一个子命令 text-replace-tool(trt)：高性能、跨平台文本替换工具，面向大型目录（300GB+）与海量文件，支持流式并行处理、二进制检测、结构化 ZIP 备份、一键撤销、正则与大小写控制。"

## Clarifications

### Session 2026-06-01

- Q: 执行 `-u` 撤销时如何确定要还原哪个备份（尤其多个目标目录共享同一同级 `backup/` 时）？ → A: 按目标目录定位——备份内记录其来源目录元数据，`-u` 只还原与所给 `-d` 匹配的最近一次备份。
- Q: 替换写回文件时如何保证中途崩溃不产生半写损坏文件？ → A: 临时文件 + 原子改名——先写入同目录临时文件，再原子 rename 覆盖原文件。
- Q: 单个文本文件如何处理替换以兼顾内存效率与超大单文件？ → A: 阈值混合——小文件整体载入替换，超过阈值的大文件用带重叠窗口的流式替换。
- Q: 启用正则（`-r 1`）时替换文本 `-n` 是否支持捕获组引用？ → A: 支持捕获组——`-n` 中可使用 `$1`、`${name}` 等引用，未匹配的组按空串处理。
- Q: 新增"批量替换"需求具体指哪种能力？ → A: 多组替换规则——一次运行传入多组"旧→新"，对目录内文件同时应用全部规则。
- Q: 多组替换规则通过什么方式提供？ → A: 规则文件——通过 `--rules <文件>` 提供，每行一组（旧/新由分隔符分隔）。
- Q: 多条规则同时命中同一段文本时如何应用？ → A: 单遍同时匹配——扫描一遍文本，每个位置只被匹配一次，重叠时按规则先后顺序优先，避免级联替换。
- Q: 正则/大小写等开关对规则统一生效还是可逐条设置？ → A: 全局统一——`-r`、`-c` 等开关对本次运行的所有规则统一生效。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在目录中批量替换文本 (Priority: P1)

使用者希望对某个目录下的所有文本文件执行一次"查找并替换"，把指定的旧文本替换为新文本，
即使目录非常庞大（数百 GB、海量小文件）也能正常完成且不破坏二进制文件。

**Why this priority**: 这是整个工具的核心价值。没有它，工具不具备任何意义；
仅实现这一条即可作为可用的最小产品（MVP）交付。

**Independent Test**: 准备一个含若干文本文件与少量二进制文件的目录，运行替换命令，
验证文本文件中的目标内容被正确替换、二进制文件保持原样、命令报告被修改的文件数量。

**Acceptance Scenarios**:

1. **Given** 目录中有 3 个文本文件包含字符串 "foo"，**When** 执行将 "foo" 替换为 "bar" 的命令，
   **Then** 这 3 个文件中的 "foo" 全部变为 "bar"，且命令报告"已修改 3 个文件"。
2. **Given** 目录中混有文本文件与图片/可执行等二进制文件，**When** 执行替换，
   **Then** 仅文本文件被处理，二进制文件被自动跳过且内容字节不变。
3. **Given** 目标文本在目录中不存在，**When** 执行替换，
   **Then** 无任何文件被修改，命令以成功状态结束并报告"已修改 0 个文件"。
4. **Given** 一个超大目录（数百 GB），**When** 执行替换，
   **Then** 进度信息在数秒内开始反馈，进程内存占用不随文件数量增长而显著上升。
5. **Given** 通过 `--rules` 提供了多组替换规则（如 "foo→bar"、"baz→qux"），**When** 执行替换，
   **Then** 目录内文件中两组规则同时被应用，且对同一位置每条规则至多匹配一次（无级联替换）。

---

### User Story 2 - 备份与一键撤销 (Priority: P2)

使用者担心替换出错，希望在替换前自动备份将被修改的文件，并能在替换后用一条命令撤销上一次操作、
恢复到替换前的状态。

**Why this priority**: 替换是破坏性操作，安全网（备份+撤销）显著降低误操作成本，是核心功能之外最重要的保障；
但它依赖 US1 先产生替换动作，故列为 P2。

**Independent Test**: 在启用备份的情况下执行一次替换，确认生成了备份归档；随后执行撤销命令，
验证目录恢复到替换前内容，且对应备份被自动清理。

**Acceptance Scenarios**:

1. **Given** 备份处于启用状态，**When** 执行一次产生修改的替换，
   **Then** 在目标目录同级的 `backup` 目录下生成一个以时间戳命名的 ZIP 备份，且仅包含被修改过的文件。
2. **Given** 备份 ZIP 内含被修改文件，**When** 查看其内部结构，
   **Then** 内部路径与原始目录层级完全一致（如 `sub/folder/file.txt`），可解压后直接覆盖还原。
3. **Given** 刚完成一次带备份的替换，**When** 执行撤销命令，
   **Then** 目录内容恢复到替换前状态，且本次使用的备份归档被自动删除。
4. **Given** 备份被显式禁用，**When** 执行替换，
   **Then** 不生成任何备份归档，并且后续撤销操作因无可用备份而给出明确提示。

---

### User Story 3 - 主命令探索与帮助 (Priority: P3)

使用者初次使用或忘记用法时，希望仅运行主程序就能看到有哪些子命令及其用途，并能查看标准帮助。

**Why this priority**: 提升可发现性与易用性，但不影响核心替换功能能否完成，故列为 P3。

**Independent Test**: 分别运行 `主命令 -ls`、`主命令 -h`、以及不带任何子命令的主命令，
验证三种情况都输出了预期内容。

**Acceptance Scenarios**:

1. **Given** 已安装本工具，**When** 运行 `主命令 -ls`，
   **Then** 输出所有可用子命令的名称及其简要描述（如 `trt  文本替换工具`）。
2. **Given** 已安装本工具，**When** 运行 `主命令 -h`，
   **Then** 输出标准用法帮助（程序用途、用法格式、可用子命令、全局选项）。
3. **Given** 已安装本工具，**When** 不带任何子命令直接运行主命令，
   **Then** 其行为等同于 `-ls`，列出子命令清单。

### Edge Cases

- 指定的目录不存在或无读写权限时：命令应以非零退出码结束并给出清晰错误说明，不产生部分破坏。
- 旧文本为空字符串时：应被视为非法输入并拒绝执行（避免无意义或破坏性替换）。
- 规则文件不存在、无法读取或全部为空行/注释（无有效规则）时：视为非法输入并以非零退出码报错。
- 规则文件中某行格式非法（缺少分隔符）时：报错并指明出错行号，不执行部分替换。
- 多组规则中存在重复或相互重叠的旧文本时：按规则在规则集中的先后顺序优先匹配，后续重叠规则在该位置不再生效。
- 撤销模式下未提供旧/新文本：允许，旧/新文本在撤销模式中为可选项。
- 同一目录连续多次替换产生多个备份时：撤销按时间从最近到最早逐次回退（每次撤销恢复并清理与该目录匹配的最近一个备份）。
- 多个目标目录共享同一同级 `backup/` 目录时：撤销依据备份内记录的来源目录元数据，仅回退与所给 `-d` 匹配的备份，互不干扰。
- 替换过程中途出错（如磁盘写满）：因采用临时文件+原子改名，原文件保持完整不被破坏；已生成的备份应保留以便恢复，并向使用者报告失败位置。
- 文件采用非 UTF-8 编码或包含无效字节：按二进制处理而跳过，避免产生乱码或损坏。
- 目标目录中存在符号链接：不跟随符号链接所指向的目录，避免越界修改或循环。

## Requirements *(mandatory)*

### Functional Requirements

#### 主命令框架

- **FR-001**: 系统 MUST 以单一可执行程序对外提供功能，并通过子命令组织各项能力。
- **FR-002**: 系统 MUST 支持 `-ls` 选项，列出所有已注册子命令的名称及其简要描述。
- **FR-003**: 系统 MUST 支持 `-h`/`--help` 选项，输出标准用法帮助信息。
- **FR-004**: 当未提供任何子命令时，系统 MUST 表现得与 `-ls` 一致（列出子命令清单）。

#### trt 子命令——文本替换

- **FR-005**: 系统 MUST 提供名为 `text-replace-tool` 的子命令，并支持简写别名 `trt`。
- **FR-006**: trt MUST 接受目标目录参数 `-d`/`--directory`，支持相对路径与绝对路径，且为必填项。
- **FR-007**: trt MUST 接受被替换文本参数 `-o`/`--old-text` 与替换文本参数 `-n`/`--new-text`；
  在非撤销模式下，`-o`/`-n` 与 `--rules` 二者必须提供其一；在撤销模式下二者均为可选。
- **FR-007a**: trt MUST 支持通过 `--rules <文件>` 提供多组替换规则，实现一次运行批量应用多组"旧→新"替换；
  规则文件每行表示一组规则，旧文本与新文本之间以约定分隔符分隔，空行与注释行（以 `#` 起始）MUST 被忽略。
- **FR-007b**: 当同时提供 `--rules` 与 `-o`/`-n` 时，trt MUST 将命令行的单组 `-o`/`-n` 视为追加到规则集中的一条规则。
- **FR-008**: trt MUST 递归遍历目标目录下的全部文件，对文本文件执行替换。
- **FR-008a**: 应用多组规则时，trt MUST 采用单遍同时匹配语义：对文本扫描一遍，每个字符位置至多被一条规则匹配，
  当多条规则在同一位置重叠命中时按规则在规则集中的先后顺序优先；MUST NOT 将某条规则的替换输出再交由其他规则二次替换（即不产生级联替换）。
- **FR-009**: trt MUST 自动检测并跳过二进制文件，保证非文本数据的字节内容不被修改。
- **FR-010**: trt MUST 支持备份开关 `-b`/`--backup`（1=启用，0=禁用），默认启用（1）。
- **FR-011**: 启用备份时，trt MUST 仅对实际被修改的文件进行备份，并将其汇集为单个 ZIP 归档。
- **FR-012**: 备份归档 MUST 存放于"目标目录同级"的 `backup` 目录中，文件名格式为 `backup_yyyyMMddHHmmss.zip`。
- **FR-013**: 备份 ZIP 内部 MUST 完整保留相对于目标目录的原始目录层级，使解压后可直接覆盖还原。
- **FR-013a**: 每个备份归档 MUST 记录其来源目标目录的元数据（即本次替换所用 `-d` 的目录标识），
  以支持在多个目标目录共享同一同级 `backup/` 目录时正确区分备份归属。
- **FR-014**: trt MUST 支持撤销模式 `-u`/`--undo`（1=撤销，0=正常），默认 0；
  撤销时按目标目录定位——仅还原与所给 `-d` 匹配的最近一次备份，并在成功后删除该备份归档。
- **FR-015**: trt MUST 支持大小写敏感开关 `-c`/`--case-sensitive`（1=敏感，0=不敏感），默认 1。
- **FR-016**: trt MUST 支持正则表达式开关 `-r`/`--regex`（1=正则，0=字面），默认 0。
- **FR-016a**: 启用正则时，替换文本 `-n` MUST 支持捕获组引用（如 `$1`、`${name}`）；
  对未参与匹配的捕获组，其引用 MUST 按空字符串处理。
- **FR-016b**: `-r`（正则）、`-c`（大小写敏感）等开关 MUST 对本次运行的所有替换规则统一生效，
  不支持逐条规则单独配置开关。
- **FR-017**: trt MUST 支持进度显示开关 `--progress`（1=显示，0=隐藏），默认 1。
- **FR-018**: trt MUST 在扫描目录的同时并行处理已发现的文件，使进度反馈不必等待整个目录扫描完成。
- **FR-019**: trt MUST 采用阈值混合策略处理单个文件：小于阈值的文件可整体载入后替换；
  超过阈值的大文件 MUST 以带重叠窗口的流式方式替换，避免将整个大文件一次性载入内存且保证跨块匹配的正确性。
- **FR-019a**: 替换写回文件 MUST 采用"先写同目录临时文件、再原子改名（rename）覆盖原文件"的方式，
  确保程序中途崩溃（如磁盘写满）时不产生半写损坏的文件。
- **FR-020**: trt MUST 在 Windows 与 Linux 两个平台上保持一致的行为与可用性。
- **FR-021**: 操作结束时，trt MUST 向使用者报告处理结果摘要（如被修改的文件数量、是否生成备份/备份位置）。
- **FR-022**: 当输入非法（目录不存在/不可访问、旧文本为空、正则表达式无效等）时，
  trt MUST 以非零退出码终止并输出清晰的错误信息，且不留下部分损坏的目录。

### Key Entities

- **子命令 (Subcommand)**: 主命令下的一项功能单元，具有名称、别名与简要描述，用于 `-ls` 列举。
- **替换规则 (Replacement Rule)**: 一组"旧文本→新文本"的映射，是替换的最小单元；可由命令行 `-o`/`-n` 给出单组，
  或由 `--rules` 文件给出多组。规则在规则集中的顺序决定重叠命中时的优先级。
- **替换任务 (Replacement Task)**: 一次替换操作的输入集合，包含目标目录、一个或多个替换规则（规则集）及各项全局开关
  （备份、大小写、正则、进度、撤销）。
- **备份归档 (Backup Archive)**: 一次替换前对被改文件的快照，表现为带时间戳的单个 ZIP，
  内部保留原始目录层级，并记录其来源目标目录的元数据，是撤销操作的数据来源与归属判定依据。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 工具能够处理规模达 300GB 以上、包含海量文件的目录而不崩溃或耗尽内存。
- **SC-002**: 对于超大型目录，使用者在启动命令后数秒内即可看到进度开始反馈，而非长时间无响应。
- **SC-003**: 进程内存占用基本恒定，不随待处理文件数量的增加而显著上升。
- **SC-004**: 二进制文件被 100% 正确跳过，替换操作不会损坏任何非文本数据。
- **SC-005**: 在启用备份的前提下，撤销操作能够把目录 100% 还原到替换前的内容。
- **SC-006**: 备份 ZIP 解压后可直接覆盖目标目录完成手动还原，目录层级与原始结构完全一致。
- **SC-007**: 同一套命令在 Windows 与 Linux 上均可成功运行并得到一致结果。
- **SC-008**: 使用者无需查阅外部文档，仅通过 `-ls`/`-h` 即可了解可用子命令及其基本用法。

## Assumptions

- 主可执行程序的调用名为 `cct`（与用户示例 `cct trt ...` 保持一致）。
- "文本文件"与"二进制文件"的区分依据通用启发式判断（如是否包含空字节等），无需使用者预先指定文件类型。
- 文本文件默认以 UTF-8 处理；无法按文本解码的文件被归类为二进制并跳过。
- 撤销为单向、按时间回退的逐次操作：每次撤销针对当前最新的一个备份归档；无可用备份时给出提示而非报错中断。
- 备份目录位于"目标目录的同级"位置，即与目标目录共享同一父目录。
- 开关类参数统一采用 `1/0` 取值约定，以贴合用户给出的参数表。
- 替换在文件内执行全部匹配项的替换（全局替换），而非仅首个匹配。
- 批量规则文件采用按行解析：每行一组规则，旧/新文本以约定分隔符（如制表符）分隔；以 `#` 起始的行与空行被忽略。
- 多组规则采用"单遍同时匹配"语义，重叠命中按规则先后顺序优先，规则间不产生级联替换。
- 正则、大小写等开关对一次运行内的所有规则统一生效。
- 本期范围仅包含主命令框架与 `trt` 一个子命令；其他子命令为后续迭代，不在本规格内。
